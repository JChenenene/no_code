# 适用场景

该模式库适用于后台系统中的抽屉表单、确认弹层、详情侧滑面板、分步弹窗和轻量向导。它适合解决“哪些任务应该在当前页完成，哪些应该打开覆盖层，如何避免频繁跳页打断上下文”的问题，常见于项目编辑、成员邀请、账单确认、快速创建和详情预览等场景。

# 页面目标

覆盖层模式的目标是让用户在不离开当前上下文的前提下完成次级任务，同时保持主页面状态和操作节奏。抽屉适合编辑或补充信息，弹层适合确认与短任务，步骤条适合分阶段引导。选择错误的承载方式，会导致用户在复杂流程中迷失，或者觉得系统层级过深、状态难以恢复。

# 推荐信息架构

推荐规则是：短确认用 Modal，详情查看或中等复杂编辑用 Drawer，多步设置或首次关键流程用 Stepper。抽屉内部结构通常为标题区、正文区、辅助说明、底部操作栏；弹层结构则更聚焦标题、说明、主次按钮。步骤条若放在覆盖层中，应始终显示当前位置和剩余步骤，且每一步信息量保持克制。

# 必备模块

- 标题区：任务名称、上下文对象、关闭入口
- 正文区：表单、说明、状态、详情或预览
- 操作栏：取消、确认、保存、下一步、上一步
- 状态反馈：提交中、保存成功、校验失败、关闭确认
- 上下文保持：关闭后回到原列表、保留筛选与滚动位置
- 风险提示：危险操作说明、不可逆影响、依赖条件

# 视觉风格与交互重点

覆盖层要与主页面形成层级差，但不能抢走所有视觉焦点。抽屉通常适合较轻遮罩和稳定分栏，弹层则应更集中，按钮权重明确。交互重点在打开关闭动画、焦点管理、Esc 退出、点击遮罩行为、未保存提醒和多步流转。若抽屉内表单较长，底部操作栏建议吸附，避免用户滚动很久后找不到提交按钮。

# 文案语气与禁忌

覆盖层文案要明确告诉用户“正在处理什么对象、执行后会发生什么”。确认弹层不要只写“确定吗”，而应说明后果，例如“移除成员后其将失去当前项目访问权限”。步骤条标题应使用任务语义，如“选择模板”“填写基础信息”“确认发布”，避免抽象命名。不要把模态框写成另一页营销长文，也不要在抽屉里堆过多与当前动作无关的信息。

# 响应式与实现边界

移动端覆盖层通常更适合全屏面板或底部抽屉，而不是桌面式居中弹窗。步骤流在小屏应保留标题、当前进度和主要按钮，避免同时展示过多辅助信息。若是纯前端演示状态，也应完整呈现打开、关闭、校验、确认和返回主界面的行为，而不只是静态截图式界面。
