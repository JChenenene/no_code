# 适用场景

该模板适用于用户个人中心、账户设置页、通知偏好页、安全设置页和品牌偏好设置页。它经常出现在后台产品或会员系统中，用来集中承接头像昵称修改、密码安全、通知开关、主题偏好、API Key 管理等个人层面的配置操作。

# 页面目标

设置页的目标是让用户快速找到需要修改的配置项，并对修改结果有稳定预期。与创建向导不同，设置页更强调“可回看、可反复修改、分类清楚”。页面应避免把所有配置堆进一个超长表单，而是通过分组、标签页或侧边导航让用户理解信息边界，降低在安全设置和普通偏好之间混淆的概率。

# 推荐信息架构

推荐结构为：顶部账户概览、左侧设置导航或顶部标签、右侧具体设置区、底部保存反馈区。常见分组包括个人资料、安全与登录、通知偏好、外观主题、开发者密钥或集成设置。若某一类配置风险较高，例如 API Key、密码重置、双因素认证，应单独形成独立模块，并提供清晰说明。

# 必备模块

- 账户概览：头像、姓名、邮箱、账号状态
- 基础资料：昵称、公司、职位、简介、语言偏好
- 安全设置：密码修改、登录设备、二次验证、会话管理
- 通知偏好：邮件、短信、系统内通知、频率设置
- 开发者配置：API Key、Webhook、令牌说明与创建入口
- 保存反馈：变更提示、成功消息、错误提醒、撤销路径

# 视觉风格与交互重点

设置页要以清晰、可信和易读为先。布局不必炫技，重点在组块分层、标题层级和字段状态统一。交互重点在表单变更检测、未保存提示、危险设置确认、复制密钥、显示/隐藏敏感信息和设备管理。安全相关模块应通过背景色、提示框或图标适度增强风险感知，但不宜造成恐吓式体验。

# 文案语气与禁忌

设置页文案应简洁明确，解释的是“这个选项会影响什么”，而不是泛泛描述功能。像“关闭后将不再接收系统异常提醒”“重置后旧密钥将立即失效”这类说明很有必要。避免使用模糊的标签名和省略条件的危险操作按钮，也不要把提示写成技术白皮书式大段说明。

# 响应式与实现边界

移动端设置页适合改为一级列表加详情页结构，避免在小屏下同时显示导航和表单。安全设置、密钥管理和设备列表应确保触控区域清晰、复制反馈明确。若当前是纯前端演示页，可使用前端状态管理模拟保存与校验，但高风险操作仍要有完整的确认和成功提示路径。
