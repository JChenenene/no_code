# 适用场景

案例展示页适用于 SaaS 官网中的案例中心、成功故事页、作品集页面和解决方案落地展示页。它的主要价值是补充“可信度”，通过具体项目或场景证明产品并非只有概念。

# 页面目标

案例页的目标是帮助用户快速看到产品被谁使用、解决了什么问题、带来了什么结果。它不需要像完整白皮书那样冗长，但每一个案例卡片都应至少说明行业背景、核心诉求和结果收益，让用户能把自己代入相似场景。

# 推荐信息架构

推荐结构为：案例总览首屏、筛选或标签区、案例卡片列表、精选深度案例、最终 CTA。若案例数量有限，可采用“3 个精选案例 + 1 个方法论总结”；若案例较多，可加入行业、页面类型、业务目标等筛选维度。

# 必备模块

- 首屏：案例价值说明、服务范围、CTA
- 标签筛选：行业、站点类型、目标场景
- 案例卡片：项目名称、行业、摘要、关键结果
- 深度案例：问题背景、方案结构、上线结果
- 方法论补充：共性做法、适配流程、交付特点
- 结尾 CTA：预约演示、查看更多、开始创建

# 视觉与交互重点

案例页适合使用更有内容感的版式，卡片、截图、指标数字和标签需要共同构成层级。卡片不能全部做成同一个大小和权重，精选案例应明显更突出。交互重点在筛选、卡片 hover、高亮结果数字和详情展开，但要保证不点开也能看懂每个案例的核心价值。

# 文案语气与禁忌

案例文案需要具体、真实、简洁，优先描述“原先遇到什么问题、最终做成了什么、效率或结果发生了什么变化”。避免写成纯宣传稿，也不要使用无法验证的浮夸结论。指标可以适度模拟，但要符合实际业务语言，例如“上线时间缩短至 2 天内”“首屏转化率提升趋势明显”。

# 响应式与实现边界

移动端案例页要优先保留标签筛选、案例摘要和关键结果，复杂横向卡片改为纵向堆叠。图片预览区和指标区需要保持清晰，不要缩成不可读的缩略图。若没有真实案例图片，可使用统一风格的占位截图或结构示意图，但文本必须像真实案例，而不是空洞模板。
