# 适用场景

该模板适用于产品功能详情页、解决方案介绍页、竞品差异说明页和“为什么选择我们”类对比页面。它通常位于官网中后段，用来承接用户已经产生兴趣、开始深挖能力边界与差异点的阶段，适合详细讲清某个核心能力或某组方案组合。

# 页面目标

功能详情页的目标是把抽象能力翻译成可理解的业务收益，并帮助用户判断该能力是否适合自己的场景。若是方案对比页，则还要明确不同版本、不同产品线或不同路径之间的差异。页面不能只列堆功能名词，也不能变成无边界长文，而是要让用户在滚动过程中逐步形成“这项能力是什么、解决了什么、为什么优于替代方案”的稳定判断。

# 推荐信息架构

推荐结构为：功能首屏、价值拆解、适用对象、关键流程或界面演示、与替代方案对比、案例或结果证明、FAQ、行动区。若页内包含多个方案，可在首屏后加入方案切换或锚点导航。对比区应围绕用户决策点展开，例如效率、可控性、部署方式、品牌一致性、协作流程，而不是为了“显得全面”做泛泛的大表格。

# 必备模块

- 首屏区：功能定位、主要收益、CTA
- 价值拆解：按问题到结果的逻辑说明能力
- 适用对象：行业、团队角色、典型使用条件
- 演示区：流程图、界面截图、前后对比或关键步骤
- 对比区：与传统方式、竞品方式或其他方案的差异
- 案例证明：结果数字、客户摘要、落地场景
- FAQ：上线周期、技术要求、定制边界、收费方式

# 视觉风格与交互重点

功能详情页比首页更偏信息型，但仍需保留品牌感。视觉重点在标题层级、对比模块、流程演示和数据证明区。交互可适度加入选项卡切换、对比高亮、流程步骤 hover 或截图放大，但不要把说明页做成复杂应用。对比区尤其要保持可读，不能让表格又长又密导致用户根本不愿看。

# 文案语气与禁忌

文案应以“问题 - 方法 - 结果”逻辑推进，尽量用场景化表达，例如“当团队需要在多品牌项目中统一交付标准时……”。避免大量技术黑话和无结果的自夸式语句。对比描述要克制，强调自己的优势即可，不建议用贬低式文案攻击其他方案。

# 响应式与实现边界

移动端功能详情页应保留最关键的价值说明、核心对比点和主 CTA，复杂对比表可改为分段卡片或横向滑动模块。流程演示区在小屏下要避免过长截图堆叠，可拆为步骤卡。若当前为演示页，没有真实视频或截图时，也应使用统一风格的占位界面示意图，保证说明完整性。
