# 适用场景

该模板适用于产品帮助中心、文档首页、FAQ 聚合页、开发者接入文档和售后支持中心。它也可被理解为“文档中心”“帮助文档页”“问题解答页”或“支持中心首页”。它既可用于营销站点中的支持板块，也可用于后台系统中的帮助入口，核心任务是让用户快速自助找到答案，而不是把所有内容都堆成知识仓库目录。

# 页面目标

帮助中心页的目标是缩短“遇到问题”到“找到可执行答案”的距离。高质量帮助页应同时照顾两类用户：一类想直接搜索问题，另一类需要按主题逐层浏览。页面需要明确展示常见入口、热门问题、分类导航和联系支持的下一步，避免用户在内容过多的长列表中迷失方向。

# 推荐信息架构

推荐结构为：搜索区、热门问题区、文档分类区、精选教程或快速开始、支持入口、页脚补充。若内容面向开发者，可将“快速开始”“认证方式”“错误码”“SDK 示例”等放在更靠前位置；若面向业务用户，则优先展示“创建应用”“账户与账单”“成员协作”“部署与发布”这类主题。分类数量不宜过多，避免用户一屏看到十几个并列入口却无法判断从哪里开始。

# 必备模块

- 搜索区：大搜索框、热门关键词、输入联想
- 常见问题：高频问题列表、跳转详情、更新日期
- 分类导航：快速开始、功能使用、部署、账单、权限、故障排查
- 精选教程：分步骤指南、视频或图示入口、入门案例
- 支持入口：工单、邮箱、客服、社区、状态页
- 反馈区：文章是否有帮助、意见提交、相关文档推荐

# 视觉风格与交互重点

帮助中心更需要秩序和可扫描性，而不是营销式冲击力。搜索区可适当放大，分类卡片和问题列表应具备明确层级，图标使用保持一致。交互重点在搜索联想、分类切换、目录锚点、文档展开收起和“相关文章推荐”。若页面支持文档正文阅读，目录和正文的视觉关系要清楚，避免用户滚动很久却不知道当前位置。

# 文案语气与禁忌

帮助文案应以解决问题为导向，标题要直接对应用户意图，如“如何邀请团队成员”“如何查看部署日志”“如何升级订阅”。避免把帮助中心写成品牌宣传页，也不要把问题标题写得像营销广告。操作步骤和注意事项应分开表达，错误场景和前置条件要说明清楚。

# 响应式与实现边界

移动端帮助页应优先保留搜索、热门问题和分类入口，长目录与侧边导航可折叠。文档正文页在小屏下需要清晰的标题层级、合适的代码块滚动区域和返回上级入口。若当前无真实文档系统，可用静态策展内容与假搜索结果演示结构，但整体导航和阅读体验必须完整。
