# 适用场景

该模板适用于多成员协作系统、企业后台、AI 应用平台中的团队设置页、成员管理页、角色权限页和邀请协作页。它也常被用户直接称为“团队成员管理页”“权限管理页”“邀请成员页”或“协作成员设置页”。它适合解决“谁可以进入系统、谁能操作哪些资源、如何邀请和移除成员”的管理问题，通常属于后台设置体系中的核心页面。

# 页面目标

团队与权限页的目标是帮助管理员快速理解当前组织结构、成员角色分布和权限边界，并能安全地进行邀请、调整和移除操作。页面不仅要展示成员列表，更要降低权限错误配置的风险。理想状态是：管理员能一眼看出谁是所有者、谁拥有管理权限、哪些邀请待处理，以及某个操作会影响哪些范围。

# 推荐信息架构

推荐结构为：组织概览、成员列表、角色说明、邀请入口、权限明细或角色配置区。若系统权限模型简单，可将角色说明嵌入列表顶部；若权限粒度较复杂，可拆为“成员管理”和“角色权限”两个区域或标签页。成员列表应支持搜索、筛选角色、查看状态和快速操作，邀请流程则应保持尽可能短。

# 必备模块

- 团队概览：团队名、成员数、剩余席位、所有者信息
- 成员列表：头像、姓名、邮箱、角色、状态、最近活跃
- 邀请入口：输入邮箱、选择角色、发送邀请、查看邀请状态
- 角色说明：所有者、管理员、编辑者、查看者等权限差异
- 权限操作：修改角色、撤销邀请、移除成员、转移所有权
- 风险提示：高权限变更、席位不足、操作影响范围说明

# 视觉风格与交互重点

团队页更适合稳重、结构清晰的后台设计。成员列表和角色标签需要明显区分，但不宜使用过多饱和色。交互重点在邀请弹层、角色切换下拉、危险操作确认、批量筛选和状态提示。对于“移除成员”“转移所有权”等高风险行为，必须有二次确认和解释性文案，避免误操作造成不可逆影响。

# 文案语气与禁忌

成员与权限文案应准确表达权限边界，例如“可管理项目与成员”“仅可查看部署结果”“可编辑但不可删除团队”。避免将角色名称取得过于花哨，也不要让权限解释藏在用户很难发现的提示里。邀请提示要清楚说明成员收到什么、何时失效、未接受前处于什么状态。

# 响应式与实现边界

移动端成员管理页建议保留团队概览、成员卡片列表和邀请入口，复杂权限矩阵应转为详情页或抽屉。角色切换操作在小屏下要避免误触，危险操作建议放入二级菜单。若演示态没有真实组织服务，可使用不同角色和邀请状态的模拟数据，但必须保留状态变化与提示反馈。
